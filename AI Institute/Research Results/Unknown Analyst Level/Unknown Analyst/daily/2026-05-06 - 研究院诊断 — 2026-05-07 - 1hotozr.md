## 4. 推荐优化（按预期影响排序）

### 优化 1：白板 handoff 默认收件人兜底
- **Target**：`whiteboard:card.finalize`（或负责生成 handoff_recipient 字段的步骤模板）。
- **Suggested change**：在 finalize 步骤中加入校验——若 `handoff_recipient` 为空且卡片包含可识别的 sector/topic 标签，则按标签映射到默认 owner（例如 TMT 关键词 → tmt-analyst，能源 → energy-analyst）；若仍无法确定，至少 fallback 到 `chief-strategist` 而非 `(none)`，并在产物里标注 `routing_fallback=true`，便于后续审计。
- **Rationale**：§3.2 显示 25/109 ≈ 23% 卡片没有收件人，是最大的链路断点。
- **Risk**：错配（把无关卡硬塞给某个分析师）会增加误派噪声；需在 fallback 标签上加 `routing_fallback=true` 给操作员留追溯口。

### 优化 2：Pending-runs 清扫器 + 终结回调审计
- **Target**：系统组件 `workflow.runtime.finalizer`（或诊断师自身在每日开始时的清理钩子）。
- **Suggested change**：增加每日扫描——对超过 2 小时仍 pending 的 run，强制查询其 brain task 状态，若 brain 已完成但 run 未终结，则调用回写终结；若 brain 仍在 running，记录 stale 标签并告警。在 fleet-health 输出里增加 `pending_runs_age_buckets`（<2h / 2-12h / >12h）以暴露真实尾巴。
- **Rationale**：§3.1 中 26 条 parked，如果不清扫会逐日累加，最终淹没失败信号。
- **Risk**：误终结仍在跑的 brain task 会丢结果；必须以"brain 状态查询为先、强制终结为后"的顺序，且只对 brain 已 done 的 run 终结。

### 优化 3：缓解 tmt-analyst 单点过载
- **Target**：`task_template`：daily-report-editor 派发 prompt + 各 topic_pitch_* 的 follow-up 路由段。
- **Suggested change**：在派发模板中加一条软上限——单角色单窗口收件 ≥10 时，将后续 TMT 相关 handoff 改为 cc 形式或拆分至 thematic-researcher / sector-specific 子分析师，避免单一收件人成为吞吐瓶颈；同时在 prompt 中写明"避免把跨题材综合材料默认归到 TMT"。
- **Rationale**：§3.3 中 tmt-analyst 净入站 23、净出站 1，是最严重的负载倾斜点。
- **Risk**：分流会让原本由 TMT 一致处理的话题被多人接手，可能损失主题连续性；建议先观测一周再固化软上限。
