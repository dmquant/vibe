## 4. 推荐优化（按预期影响排序，限 3 项）

### 优化 1：抑制白板 QA / handoff 步骤对 reviewer 的过度路由
- **Target**：白板 QA 推荐 / handoff suggestion step（具体 step 在白板会话模板中，建议定位至 `whiteboard:qa_handoff_suggest` 或同义 step；如不存在请按 task_template 找最接近的 handoff-recipient prompt）。
- **Suggested change**：在 prompt 中加入显式约束——"reviewer 类 recipient（chief-risk、qa-manager、compliance-officer）仅在卡片明确包含风险触发条件（破位、合规事件、对手方暴露、风控阈值跨越）时方可作为 handoff_recipient；默认情况下应优先 primary analyst（china-macro、global-macro、sector-* 、equity-strategy、chief-strategist）"。同时要求模型在选择 reviewer 时**必须**附一句 ≤30 字的风险触发理由，便于事后审计。
- **Rationale**：§3.1 显示 chief-risk 是除 none 外白板第一接收人（17），明显违背"reviewer 仅作为 escalation 末端"的设计初衷。
- **Risk**：过严的硬性约束可能让真实风控线索漏掉 escalation；因此采用"默认抑制+理由触发"而非完全禁用。

### 优化 2：在白板 closeout 模板中强制 recipient 二选一
- **Target**：`whiteboard:card_closeout` / 卡片完成模板（请按白板模板命名校准）。
- **Suggested change**：将 handoff_recipient 字段从可选改为半强制——若分析师选择 `none`，必须在同一字段附加 `terminal_reason`（枚举：`question_resolved` / `data_dead_end` / `merged_into_other_card` / `monitoring_only`）；否则模板拒绝提交。
- **Rationale**：§3.2 中 20 张 `(none)` 占 ~10%，无法分辨"合理终结"与"漏填"；引入 terminal_reason 让操作员能够事后过滤"漏填"型断链。
- **Risk**：增加分析师一次填写成本；若枚举值设计过窄，模型可能强行套一个不贴切的理由，反而污染审计信号。建议保留 `other:<free_text>` 出口。

### 优化 3：让诊断快照直接暴露 parked 运行的年龄分布
- **Target**：fleet-health snapshot 生成器（`/api/fleet-health/latest.md` 背后的 D1 查询，非模型 prompt）。
- **Suggested change**：在 `Pending runs` 段加入 `parked_age_buckets`（<1h / 1–2h / 2–6h / 6–24h / >24h）和最老 3 条的 `run_id` + `task_key`。
- **Rationale**：§3.3 现状只能看到 12 这个总量，无法判断是否符合"<2h 应当回填"的 SLA。补齐之后诊断师能直接判定是否触发告警。
- **Risk**：snapshot 体积上升、读写成本略增；数据敏感性低，可控。
