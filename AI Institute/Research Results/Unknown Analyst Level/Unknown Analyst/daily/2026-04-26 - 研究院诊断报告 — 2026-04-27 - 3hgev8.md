## 4. 推荐优化（按预期影响排序，Top 3）

### 优化 1 — 收紧 QA-step 的 reviewer 路由阈值
- **Target**: `weekly_report:qa_review`（或当前 weekly_report 工作流中负责"决定是否升级到 chief-risk / chief-strategist"的步骤；具体 step_id 请操作员核对）
- **Suggested change**: 在该 step 的 system prompt 中追加显式 gating：
  > "仅当存在以下情形之一才升级到 chief-risk/chief-strategist：(a) 单一仓位/敞口超过预设阈值，(b) 出现合规/法务旗标，(c) 与既定策略基线显著背离。其余情形优先 handoff 给主题对口的一线分析师（macro / strategy / sector / equity-strategy 等），并在卡片正文记录'未升级原因'。"
- **Rationale**: §3-1 显示 82% 的 auto-handoff 流向 reviewer，§3-2 显示白板 chief-risk 收件量是任何一线分析师的 ~2× 以上；显式 gating 是最直接的纠偏。
- **Risk**: 阈值过严会漏报真实风险事件；建议先在 prompt 中只加"显式记录原因"，不立刻收紧阈值，留一周观察 reviewer 队列与漏报抽样。

### 优化 2 — 为白板路由器补一张"主题→一线分析师"映射
- **Target**: 白板 handoff 路由组件（whiteboard handoff resolver / routing prompt）
- **Suggested change**: 在路由 prompt / 配置中加入主题到一线分析师的优先映射（industrials→industrials-analyst、credit→credit-analyst、energy→energy-analyst …），并把 chief-risk 设为"次选 / 显式触发"而非默认兜底；同时让 (none) 接收方的卡片必须带"无需 handoff"理由。
- **Rationale**: §3-2 的 chief-risk=24 与 §3-5 的 (none)=21 同时出现，提示"无法分发→默认丢给 chief-risk"或"无法分发→不分发"两种回退路径都在被触发。补映射能同时缓解两个症状。
- **Risk**: 错配主题会把卡片送错分析师；建议先以 shadow / 提议形式输出"建议 recipient"，由人工或下一轮 QA 确认 1–2 天再切硬路由。

### 优化 3 — 给 pending runs / 长 running runs 设一个 2h 的清算扫描
- **Target**: 调度层 / `system component: pending-run reaper`（如不存在则提议新建）
- **Suggested change**: 每小时扫描 pending（已停泊待终态）+ running 超过 2h 的 weekly_report 类 run，输出一个"长尾 run 列表"工件（含 run_id、停留阶段、最后 event 时间），交给操作员或 chief-of-staff 角色处理；零失败但高 in-flight 是典型的"卡在终态写入"前兆。
- **Rationale**: §3-3 显示 21 running + 12 pending 合计 33 条 vs 19 completed，而失败计数为 0，提示问题集中在终态收尾而非执行本身。
- **Risk**: 报告本身只读，风险低；唯一成本是操作员需要消化新工件。建议先以日报附录形式呈现，不接入告警通道。
