## 4. 推荐优化（按预期影响排序，Top 3）

### 优化 1：分散 TMT 落点
- **Target**：`mailbox auto-handoff routing prompt`（chief-strategist / thematic-researcher / altdata-analyst 的 handoff 决策段）
- **建议变更**：在 handoff 决策 prompt 中加入"过去 6h 内已向同一 hand 转发 ≥3 次时，请优先考虑等价二级落点（如 TMT 主题可分流到 thematic-researcher 自身、sector 级 analyst、或 chief-strategist）"。如系统层无近期路由可见性，则在 prompt 中直接列出 TMT 内容的 2–3 个备选落点池。
- **Rationale**：tmt-analyst 单点 26.5% 集中度（§3.2）；上游 thematic / altdata / sentiment 三条线全部汇入。
- **Risk**：分流后部分 handoff 可能落到准备度更低的 hand，导致下游处理质量下降；上线后需观察 1–2 个窗口的下游完成率。

### 优化 2：白板写卡强制声明下游
- **Target**：`whiteboard.card_emit` 模板（写卡阶段的 system prompt）
- **建议变更**：在写卡 schema 里把 `handoff_recipient` 改为必填，并允许显式枚举值 `__terminal__`（终结卡）；在 prompt 末尾增加"若该卡片没有下游，必须填 `__terminal__` 而不是留空"。
- **Rationale**：38/165 = 23% 卡片无 handoff（§3.3），无法判断是死端还是终结。
- **Risk**：会增加写卡 token 消耗；少数确实是终结卡的旧模板需要批量补丁。

### 优化 3：补全 workflow_kind 分类
- **Target**：workflow 注册表 / `workflow_kind` 默认值逻辑
- **建议变更**：在 workflow 注册阶段对 `workflow_kind` 设默认值 `routine`（或显式校验失败），并对当前 17 条 (none) 的 task_key 做一次性回填；同步在写入 D1 前增加非空校验。
- **Rationale**：17/38 = 45% 未声明 kind（§3.4），是元数据卫生问题，影响后续诊断维度。
- **Risk**：默认值若错配（把 ad-hoc 当成 routine），可能扭曲未来诊断；建议先回填、再启用强校验。
