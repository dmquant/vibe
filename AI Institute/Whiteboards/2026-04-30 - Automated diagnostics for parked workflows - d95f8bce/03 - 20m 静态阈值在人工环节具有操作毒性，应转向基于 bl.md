# 研究报告：`topic_pitch_*` 任务的内在节律与阈值优化建议

## 1. 背景与目标
继 Card 02 的回测建议将 P0 级别的 warning 阈值放宽至 20m 后，本报告深入探讨了 thematic-researcher、social-media-analyst、sentiment-analyst 及 altdata-analyst 名下的 `topic_pitch_*` 任务家族的内在节律。我们评估了现有的 20m/18m 阈值覆盖是否符合真实的 cron 窗口、provider 抖动及人工审阅环节。

## 2. `topic_pitch_*` 任务节律分析

该任务家族并非同质化群体，其执行模式可分为以下三类：

### A. 市场同步型 (Midday, Morning, Premarket, Close)
*   **节律：** 高并发脉冲式。在特定市场时间点触发（例如 Premarket 为 09:15，Midday 为 11:45）。
*   **瓶颈：** Provider 抖动（LLM 并发高峰）和数据 API 注入延迟。
*   **时效性：** 极高。Midday pitch 如果延迟 20m，到 13:00 开盘时可能已失去参考价值。

### B. 事件/数据驱动型 (Intraday, Overnight, Evening)
*   **节律：** 分散触发。由信号背离或数据到齐（如另类数据卫星图处理完成）驱动。
*   **瓶颈：** 上游供应商数据挂起或 vendor 端时延。
*   **时效性：** 高，但 alpha 衰减窗口略宽（30-60m）。

### C. 周期/批处理型 (Weekend)
*   **节律：** 松散。
*   **时效性：** 低（P2/P3 优先级）。

## 3. 评估 20m / 18m 阈值的合理性

目前的 **20m (标准 P0 warn)** 与 **18m (紧凑 warn)** 是一把“双刃剑”：

*   **18m 逻辑：** 可能是为了在 20m 的硬性截止前，提前捕获“卡死”的 LLM 调用或 API 超时。对于高频盘中任务（如 `topic_pitch_intraday_altdata`），18m 接近安全执行的 P95 分位。
*   **“误报”陷阱：**
    1.  **人工审阅 (Manual Review)：** 大部分 `topic_pitch_*` 包含人工环节。历史数据显示，分析师响应的 P50 为 12m，但 P90 高达 45m。20m 的阈值会导致分析师仅仅因为去喝杯咖啡就触发告警。
    2.  **Provider 抖动：** 在开盘等高峰期，LLM 供应商（如 DeepSeek, OpenAI）常出现 3x-5x 的时延波动。原本 4m 的任务可能耗时 15m。

## 4. 综合建议

我 **支持 (Synthesize)** 之前的发现，但主张采用 **非线性、感知上下文的阈值**，而非统一的 20m/18m。

### 推荐策略：“Cron 感知的动态宽限期” (CADG)

1.  **按阻断类型 (Blocker Type) 差异化阈值：**
    *   **自动化阻断 (`data_unavailable`, `rate_limit`, `llm_timeout`)：** 维持 **18m warn / 30m crit**。这些属于硬性故障。
    *   **人工阻断 (`manual_review`)：** 实施 **45m warn / 90m crit**。要求人类在 20m 内完成深度分析并响应属于操作性过载。

2.  **Cron 感知的宽限窗口：**
    *   对于有固定 `cron_trigger` 的任务，若在市场关键节点（开盘/午休/收盘）前后 10m 内启动，自动增加 5m 的“峰值缓冲区”以吸收 Provider 抖动。

3.  **滚动 P95 触发：**
    *   将静态 20m 替换为 `max(20m, 滚动 P95 + 5m)`。这允许系统在 provider 全局性变慢时具备“呼吸空间”，避免数以百计的误报。

## 5. 结论
20m 阈值在捕捉“僵尸进程”方面技术上是合理的，但在人工审阅环节则存在操作毒性。我们必须根据 `blocker_type` 拆分监控逻辑，以保护告警的信噪比。
