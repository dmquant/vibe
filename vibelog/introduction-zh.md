# AI 股票研究院 · 项目完整介绍

*面向投资经理、配置机构、研究负责人的项目导读*

---

## 一句话概括

一座由**专业 AI 分析师**驻席的数字化股票研究院 —— 像卖方研究所一样、按时、成
体系地产出晨会简报、每日日报、个股深度研究、投委会纪要，且**每一张报告都可
追溯回最初证据**。

---

## 为何存在

传统研究所面临四个硬约束：

- **覆盖受限于编制。** 想覆盖更多标的 = 招更多分析师 = 更高人力成本。
- **交付以日计。** 晨会简报难有深度；深度研究总是姗姗来迟。
- **质量依赖个人。** 明星分析师一走，机构记忆随之蒸发。
- **决策难以复盘。** 昨天的论点今天变成未被追踪的假设。

投资人常常要在**速度与深度**、**广度与专精**之间做取舍。本项目把这一取舍移
除：一支由专业 AI 分析师组成的团队，在结构化工作流下协同 —— 覆盖可横向扩
展、节奏由机器驱动、每一条结论都留有完整证据链。

除五大旗舰工作流外，研究院还为每位分析师配置了**专属定时任务** —— 例如「首席
经济学家 · 晨间宏观简报」每工作日 07:00 执行、「首席量化师 · 因子表现日报」
16:00、「信用分析师 · 利差监控」17:00 —— 它们为主要工作流提供**新鲜证据**，
使晨会与日报不再从零开始。

**周度研究周报**有两点独特设计。其一是**逐分析师**的：39 位分析师每位都有专属
周报，按哈希分配到周日 24 个整点中的某一个（周日 00:00–23:00 SG，每个时段约
1–2 位分析师）。规划阶段把覆盖窗口锁定为**本周周一到周五**（例如周日 2026-04-26
覆盖周一 2026-04-20 到周五 2026-04-24），整条流水线四个阶段共享同一日期口径。
其二是**多模型流水线**：每份周报由三种模型分阶段生成，再由 QA Manager 审阅：

## 研究白板 · 多分析师实时辩论

前述四类产品都遵循**确定性模板**：相同输入产出相同结构。**研究白板**是它们
的创造性对立面 —— 一个开放的**辩论室**，AI 分析师在其中就某一议题实时协同
研究，一张卡片扩展、质疑或重构前一张。

**白板与工作流的关键差异：**

- 白板线程**不使用模板**。开局由**模型自己**在分析师名单中挑选一位分析师、
  同时从该分析师领域内选择一个**有时效性的研究议题** —— 基于实时网络搜索
  与模型自身判断，不预设问题。
- 每张"卡片"是一位分析师的完整研究报告 + 一份**结构化交接**：由谁接力、
  研究什么、以何种立场。
- 每张卡片都必须**声明立场**（stance），即对上一张卡片的态度：

```
  ╔═══════════════╤══════════════════════════════════════════════════════╗
  ║ Stance 立场   │ 分析师的任务                                         ║
  ╠═══════════════╪══════════════════════════════════════════════════════╣
  ║ initial       │ 开启线程，提出主论点                                  ║
  ║ support       │ 用新证据扩展前一论点                                  ║
  ║ deny          │ 挑战前一论点，揭示被忽视的风险                        ║
  ║ stress-test   │ 压力测试前一论点的脆弱假设                            ║
  ║ synthesize    │ 综合分歧观点，形成统一视角                            ║
  ║ stop          │ 结束线程；最后一张卡片                                ║
  ╚═══════════════╧══════════════════════════════════════════════════════╝
```

- 线程默认运行 **10 张卡片**（可扩容）。模型根据下一个未回答的问题，自动
  挑选最适合的下一位分析师。

**一条典型线程长什么样：**

```
  Card 01 │ TMT 行业分析师 · initial
  ━━━━━━━━┼──────────────────────────────────────────────────────────────
          │ TOPIC   AI 资本开支瓶颈换挡：从 GPU 短缺到并网电力短缺
          │ THESIS  超大规模厂商约束从"芯片供给"切换到"并网电力"，
          │         定价权在迁移。
          │ → 交接：公用事业分析师 · synthesize
          ▼
  Card 02 │ 公用事业分析师 · synthesize
  ━━━━━━━━┼──────────────────────────────────────────────────────────────
          │ 在数据中心自备电源、IPP 签长期 PPA、监管公用事业扩张 rate
          │ base 三条路径下，给出未来 12–18 月受益者排序。
          │ → 交接：首席风控官 · stress-test
          ▼
  Card 03 │ 首席风控官 · stress-test
  ━━━━━━━━┼──────────────────────────────────────────────────────────────
          │ 压力测试新并网规则与费率分摊变化下的公用事业/IPP 盈利下修风
          │ 险；给出红/黄/绿风险灯。
          │ → 交接：外汇策略师 · synthesize
          ▼
   ...    │ (最多 10 张卡片)
```

**为什么重要：**

- **每条推理都是完整链条**：论点 → 质疑 → 压力测试 → 综合。没有分析师拥有
  "最后发言权"；没有论点未经检验。
- **交接是一等公民**：每张卡片**既指定下一位分析师，也说明理由**。交接链条
  本身就是可审计的研究元数据。
- **覆盖通过组合扩展**：一条线程可以在一小时内**跨八个专业领域**展开 ——
  传统研究所要召开一场正式会议才能做到的事。
- **每小时自动开局**：每到整点系统自动启动一条新线程 —— 模型自选分析师、
  自选议题；用户也可在前端手动启动。

**内置弹性。** 如果被调度的模型繁忙或某台边缘节点离线，系统会换一位分析师
重试；若全部边缘节点失败，会回落到**云端 API 模型**（无需边缘硬件）。超出
一轮轮询预算的任务由后台收割器接管，在下一个五分钟周期内完成。**线程实际
上无法卡死**。

## 研究工作如何组织

研究院使用三种协作原语：

**Workflow（工作流）**：结构化多步骤配方，按顺序串联多位分析师。晨会简报 5
步数据采集 + 1 步合成；深度研究 12 阶段 SOP。工作流是**确定性**的：相同输入 →
相同结构，产出达到**审计级别**。

**Whiteboard（研究白板）**：分析师之间的即兴协同研究。一条线程由一位分析师
选题启动，然后接力给同事，每张"卡片"都对前一结论进行**支持、否定、压力测试
或综合**。整条线程是一份完整的推理链，含明确的交接记录。

**Mailbox（内部邮件）**：分析师之间的结构化请求-响应。投委会主席要求风险分
析师压测某情景时，发出一封邮件；请求变成一个有交付物、有时间戳回复的任务。

三者共享同一基础设施：每位分析师写入个人云端工作空间；每份产出存储一次、引
用多次；研究院**全部历史产出可按主题 / 分析师 / 日期检索**。

## 治理与可审计性

每一次运行都**完全透明**：

- 每位分析师拥有独立工作空间文件夹；每份产出文件带路径、大小、时间戳、一键
  预览。
- 每份报告列出引用源、数据来源以及模型尝试轨迹（用于失败诊断）。
- 每份投委会纪要记录每一项论点的提出人以及主席如何收敛分歧。
- 每一次定时运行都是**可点击、可回放**的会话 —— 消息、产出、推理链全部留存。

这是可度量性的先决条件。预测准确率、因子表现、评级调整、建议命中率均汇入**
自进化学习回路**：优秀论据强化方法论，薄弱论据被审查并重新训练。

## 第三方集成（v1 API）

研究院对外暴露一组**作用域受限**的 v1 API，用于与外部系统对接 —— 例如让监控
列表自动排队深度研究、让新闻订阅自动开启白板线程、让外部仪表盘订阅事件流。

```
  ┌─────────────────────────────────────┬───────────────────┬─────────────────────────────┐
  │ 端点                                │ 所需 scope        │ 用途                        │
  ├─────────────────────────────────────┼───────────────────┼─────────────────────────────┤
  │ POST /api/v1/research/queue         │ research:write    │ 批量入队若干股票代码进入    │
  │                                     │                   │ 12 阶段 SOP 深度研究        │
  ├─────────────────────────────────────┼───────────────────┼─────────────────────────────┤
  │ POST /api/v1/whiteboard/threads     │ whiteboard:write  │ 用外部提供的议题开启一条    │
  │                                     │                   │ 白板线程                    │
  ├─────────────────────────────────────┼───────────────────┼─────────────────────────────┤
  │ GET  /api/v1/events?since=<cursor>  │ events:read       │ Stripe 风格的游标事件流，   │
  │                                     │                   │ 涵盖运行完成、卡片失败、    │
  │                                     │                   │ 邮箱投递结果等所有终态事件  │
  └─────────────────────────────────────┴───────────────────┴─────────────────────────────┘
```

API 密钥在 `/admin/keys` 页面签发，前缀 `sk_ins_`，落库前以 SHA-256 哈希存储，
签发时即绑定 scope，可一键吊销。当前可用的 7 个 scope：`research:write`、
`whiteboard:write`、`briefing:trigger`、`mailbox:write`、`sessions:read`、
`analysts:read`、`events:read`。每次调用都会更新该密钥的 `last_used_at`，
便于在管理界面识别长期闲置的密钥。

事件日志为**仅追加 + 游标读取**：第三方消费方按自己节奏轮询
`/api/v1/events?since=<last_id>`，便能拿到这个游标之后的全部新事件 —— 无需
搭建 Webhook 服务器，也无需自行实现重放逻辑。

## 内置弹性

研究院的长链工作流面对三类风险：边缘节点繁忙、模型偶尔忘记写文件、agent-route
负载均衡器把任务长时间挂在 `queued` 状态。三层各对应一个解决方案：

**多模型流水线 + 自动换手。** 周报本身就是 Gemini → Codex → Claude 三模型协
作 —— 任何一种模型的偶发问题只会影响某个阶段，不会击穿整条链路。深度研究与
白板卡片若遭遇"无可用边缘节点"或被节点 watchdog 收割，系统会换一种模型重
试；若所有边缘模型都失败，则切换到无需边缘硬件的**云端 API 模型**。

**工作空间稳健性 —— 文件抢救。** 模型偶尔会"在响应中给出正确的 markdown 内
容、却忘记真正写入文件"，导致后续步骤读不到上一步产出。两层保护：第一，每
份 prompt 都附带 *Workspace Robustness* 条款，允许下游步骤在文件缺失时回落
使用上文中的内容继续推进；第二，工作流终态后由 worker 比对**声明的输出文
件**与**会话工作空间实际内容**，对缺失文件用对应 step 的响应文本**自动补
写** —— 工作记录界面与周报预览总能看到一份完整的交付物。

**队列时长不计入超时。** 任务在 agent-route 的负载均衡队列中等待的时间不再
计入 30 分钟超时阈值。客户端的"过期判定时钟"只在任务**真正切换到
`running` 状态后**才开始计时（写入卡片元数据的 `running_since` 字段）。无论
排队多久，都不会触发误判超时。

## 自动协作

邮箱（mailbox）一直是研究院的协作层，但长期以来每封邮件要么由人工触发，要
么是工作流首次派发的副产物。现在分析师可以**自己协作**：当 routine 或 global
工作流的最后一步识别到"另一位分析师应该回答的问题"时，它会输出一段结构化的
`follow_ups` JSON；worker 解析、用 36 小时去重缓存判重，然后创建一封邮件并附
带 adhoc 派发，让接收者自动开始工作。

```
  ┌──────────────────────────────────┐
  │ 工作流终态完成                    │
  └─────────────────┬────────────────┘
                    │
                    ▼
  ┌──────────────────────────────────┐    解析 fenced JSON 块：
  │ Worker · auto-handoff 扫描器     │ ─► {"follow_ups":[
  │ （扫描最后一步的响应文本）        │      {"to":"chief-risk",
  └─────────────────┬────────────────┘       "subject":"…",
                    │                        "question":"…"}]}
                    ▼
  ┌──────────────────────────────────┐    KV 缓存：36 小时 TTL，键为
  │ 校验 + 去重 + 上限                │    sha256(sender|recipient|content)
  │ • 是否合法分析师 id?              │    上限：每次最多 3 条 follow-up
  │ • 是否给自己发?                   │    接收者必须在分析师名册中
  └─────────────────┬────────────────┘
                    ▼
  ┌──────────────────────────────────┐
  │ createMailboxThread + 派发        │ ─► /mailbox 出现一条
  │ mode = "adhoc", prompt = ?        │    [auto-handoff] 邮件，
  └─────────────────┬────────────────┘    接收方自动开始执行
                    ▼
  ┌──────────────────────────────────┐
  │ 接收方分析师自动运行并通过现有    │ ─► 回复落入原分析师的收件箱；
  │ 派发回复                         │    自动协作只走一跳
  └──────────────────────────────────┘
```

**触发位置：**
- 单步 routine 任务（如 `daily_meetup`）：那唯一一步就是发送方。
- 多步 `weekly_report`：仅由 QA Manager（第 4 步）发送。
- Global 工作流 `briefing` 与 `daily`：仅由总编合成步发送。
- 贡献者步骤与中间步骤**不发送** —— 协作每条工作流只扇出一次，绝不按步扇出。

**自治的边界：**
- 每条工作流最多 3 条 follow-up。
- 只走一跳 —— 接收方的回复**不会**再触发新的自动邮件。
- 36 小时 KV 去重，键为 `(sender, recipient, content_hash)`，避免同一问题在每个
  周期重复出现。
- 接收方必须是名册中真实存在的分析师 id；sender ≠ recipient。

当 `/mailbox` 第一次自然出现一条 `[auto-handoff]` 邮件、而无人点击时，研究院
就开始从一个工作流引擎变成一支**会自己协调的研究团队**。

## 不是什么

- **不是交易系统**。它产出研究报告，不下单。
- **不是人类 CIO 的替代品**。它是 CIO 在 08:05 开盘前读的那张研究桌。
- **不保证 Alpha**。它保证**覆盖率、节奏与可审计性** —— 纪律化投资流程必需
  的底层基础。
