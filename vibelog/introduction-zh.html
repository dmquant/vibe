AI 股票研究院

The Digital Equity-Research Institute

一座由专业 AI 分析师驻席的数字化股票研究院 —— 像卖方研究所一样按时、成体系地产出晨会简报、每日日报、 个股深度研究、投委会纪要；每个交易日自动运行， 每一张报告都可追溯回最初证据。

项目完整介绍 · 2026
01 · 为何存在

传统研究所的四个硬约束

投资人常常要在速度与深度、广度与专精之间做取舍。 本项目把这一取舍一并移除 —— 一支专业 AI 分析师团队，在结构化工作流下协同， 覆盖可横向扩展、节奏由机器驱动、每一条结论都留有完整证据链。

约束 1

覆盖受限于编制

想覆盖更多标的 = 招更多分析师 = 更高人力成本。研究所规模受限于它能负担的人力。

约束 2

交付以日计

晨会简报难有深度；深度研究总是姗姗来迟。速度是严谨的敌人。

约束 3

质量依赖个人

明星分析师一走，机构记忆随之蒸发。流程不能沉淀下来。

约束 4

决策难以复盘

昨天的论点今天变成未被追踪的假设。缺乏可靠的反馈闭环。

周报流水线 · 三种模型，一份报告

周度研究周报有两点独特设计。其一是逐分析师的：39 位分析师每位都有专属周报，按哈希分配到周日 24 个整点中的某一个（周日 00:00–23:00 SG，每个时段约 1–2 位分析师）。规划阶段把覆盖窗口锁定为 本周周一到周五（例如周日 2026-04-26 覆盖周一 2026-04-20 到周五 2026-04-24），整条流水线四个阶段共享同一日期口径。其二是多模型流水线：

阶段 1
Gemini · 规划官

锁定本周覆盖范围、提出 4–6 个分析问题、列出数据需求与图表方案。

阶段 2
Codex · 研究员

执行规划：采集数据、填表、回答每个问题并附证据。

阶段 3
Claude · 视觉编辑

读取规划与研究产出，输出含内嵌 SVG 图表与斑马表格的自包含 HTML 周报。

阶段 4
Claude · 质量主管

审查交付完整性、跨步骤一致性、表达质量；给出修订建议与 pass/重写/不合格 评级。

不同模型扮演各自最擅长的角色：Gemini 擅长 Web Search 驱动的横向梳理，Codex 擅长结构化执行与数据加工，Claude 擅长长上下文判断、可视化呈现与编辑审查。

03 · 创造性功能

研究白板 · 多分析师实时辩论

前述四类产品都遵循确定性模板：相同输入产出相同结构。 研究白板是它们的创造性对立面 —— 一个开放的辩论室， AI 分析师在其中就某一议题实时协同研究，一张卡片扩展、质疑或重构前一张。

  • 白板线程不使用模板。开局由模型自己在分析师名单中挑选一位分析师、同时从该分析师领域内选择一个有时效性的研究议题 —— 基于实时网络搜索与模型自身判断，不预设问题。
  • 每张卡片是一位分析师的完整研究报告 + 一份结构化交接：由谁接力、研究什么、以何种立场。
  • 线程默认运行 10 张卡片（可扩容）。模型根据下一个未回答的问题，自动挑选最适合的下一位分析师。

每张卡片都必须声明立场

initial开启线程，提出主论点。
support用新证据扩展前一论点。
deny挑战前一论点，揭示被忽视的风险。
stress-test压力测试前一论点的脆弱假设。
synthesize综合分歧观点，形成统一视角。
stop结束线程；最后一张卡片。

… 最多 10 张卡片，每位分析师都由模型根据下一个未回答的问题自动挑选

为什么重要

  • 每条推理都是完整链条：论点 → 质疑 → 压力测试 → 综合。没有分析师拥有"最后发言权"；没有论点未经检验。
  • 交接是一等公民：每张卡片既指定下一位分析师，也说明理由。交接链条本身就是可审计的研究元数据。
  • 覆盖通过组合扩展：一条线程可以在一小时内跨八个专业领域展开 —— 传统研究所要召开一场正式会议才能做到的事。
  • 每小时自动开局：每到整点系统自动启动一条新线程 —— 模型自选分析师、自选议题；用户也可在前端手动启动。

内置弹性。 如果被调度的模型繁忙或某台边缘节点离线， 系统会换一位分析师重试；若全部边缘节点失败，会回落到 云端 API 模型（无需边缘硬件）。超出一轮轮询预算的任务 由后台收割器接管，在下一个五分钟周期内完成。线程实际上无法卡死。

07 · 第三方集成

面向系统集成的 v1 API

研究院对外暴露一组作用域受限的 v1 API，用于与外部系统对接 —— 例如让监控列表自动排队深度研究、让新闻订阅自动开启白板线程、让外部仪表盘订阅事件流。

端点 所需 scope 用途
POST /api/v1/research/queue research:write 批量入队若干股票代码，进入 12 阶段 SOP 深度研究。
POST /api/v1/whiteboard/threads whiteboard:write 用外部提供的议题开启一条白板线程。
GET /api/v1/events?since=<cursor> events:read Stripe 风格的游标事件流，涵盖运行完成、卡片失败、邮箱投递结果等所有终态事件。

API 密钥在 /admin/keys 页面签发，前缀 sk_ins_，落库前以 SHA-256 哈希存储，签发时即绑定 scope，可一键吊销。当前可用的 7 个 scope：research:write、whiteboard:write、briefing:trigger、mailbox:write、sessions:read、analysts:read、events:read。每次调用都会更新该密钥的 last_used_at，便于在管理界面识别长期闲置的密钥。

事件日志为仅追加 + 游标读取：第三方消费方按自己节奏轮询 /api/v1/events?since=<last_id>，便能拿到这个游标之后的全部新事件 —— 无需搭建 Webhook 服务器，也无需自行实现重放逻辑。

08 · 内置弹性

长链工作流如何稳定运转

长链工作流面对三类风险：边缘节点繁忙、模型偶尔忘记写文件、agent-route 负载均衡器把任务长时间挂在 queued 状态。三层各对应一个解决方案。

多模型流水线 + 自动换手

周报本身就是 Gemini → Codex → Claude 三模型协作 —— 任何一种模型的偶发问题只会影响某个阶段，不会击穿整条链路。深度研究与白板卡片若遭遇"无可用边缘节点"或被节点 watchdog 收割，系统会换一种模型重试；若所有边缘模型都失败，则切换到无需边缘硬件的云端 API 模型。

工作空间稳健性 · 文件抢救

模型偶尔会"在响应中给出正确的 markdown 内容、却忘记真正写入文件"，导致后续步骤读不到上一步产出。两层保护：每份 prompt 都附带 Workspace Robustness 条款，允许下游步骤在文件缺失时回落到上文中的内容继续推进；工作流终态后由 worker 比对声明的输出文件与会话工作空间实际内容，对缺失文件用对应 step 的响应文本自动补写。

队列时长不计入超时

任务在 agent-route 负载均衡队列中等待的时间不再计入 30 分钟超时阈值。客户端的"过期判定时钟"只在任务真正切换到 running 状态后才开始计时（写入卡片元数据的 running_since 字段）。无论排队多久，都不会触发误判超时。

09 · 自动协作

让邮件自己触发自己

邮箱（mailbox）一直是研究院的协作层，但长期以来每封邮件要么由人工触发、要么是工作流首次派发的副产物。现在分析师可以自己协作 —— 当 routine 或 global 工作流的最后一步识别到"另一位分析师应该回答的问题"时，系统会自动发出一封邮件并附带 adhoc 派发，让接收者自动开始工作。

① 工作流终态完成

工作流在 agent-route 上终态完成。Worker 的 syncManagedScheduledRuns 在下一次 5 分钟 cron 中拾取它。

② Worker 扫描最后一步

对 routine 与 global 工作流，auto-handoff.ts 扫描最后一步的响应文本，寻找 fenced JSON 块：{"follow_ups":[{"to":"chief-risk","subject":"…","question":"…"}]}。Legacy 的 committee/research 工作流暂不参与。

③ 校验 · 去重 · 上限

接收者必须是真实分析师 id；sender ≠ recipient；每次最多 3 条；用 KV 去重，键为 sha256(sender|recipient|content)，TTL 36 小时，避免同一问题在每个周期都重复出现。

④ 邮件派发 + 接收方自动运行

createMailboxThread() 创建邮件并附带 dispatch.mode = "adhoc"；接收方自动执行问题并把回复写回原分析师的收件箱。只走一跳 —— 这条回复不会再触发新的自动邮件。

触发位置

  • 单步 routine 任务（如 daily_meetup）：那唯一一步即是发送方。
  • 多步 weekly_report：仅由 QA Manager（第 4 步）发送。
  • Global 工作流 briefing 与 daily：仅由总编合成步发送。
  • 贡献者步骤与中间步骤不发送 —— 协作每条工作流只扇出一次，绝不按步扇出。

当 /mailbox 第一次自然出现一条 [auto-handoff] 邮件、而无人点击时，研究院就开始从一个工作流引擎变成一支会自己协调的研究团队。

10 · 治理与可审计性

每一次运行都完全透明

  • 每位分析师拥有独立工作空间文件夹；每份产出文件带路径、大小、时间戳、一键预览。
  • 每份报告列出引用源、数据来源以及模型尝试轨迹（用于失败诊断）。
  • 每份投委会纪要记录每一项论点的提出人以及主席如何收敛分歧。
  • 每一次定时运行都是可点击、可回放的会话 —— 消息、产出、推理链全部留存。

这是可度量性的先决条件。预测准确率、因子表现、评级调整、建议命中率均汇入 自进化学习回路：优秀论据强化方法论，薄弱论据被审查并重新训练。

13 · 上手路径

七步抵达研究院

  1. 打开控制台，阅读"今日定时任务" —— 研究院今天自动生产了什么。
  2. /briefing —— 今晨的晨读；每 45 秒自动刷新。
  3. /daily —— 今日收盘日报。
  4. /whiteboard —— 实时推进的研究线程。
  5. /analysts/:id —— 点进任意分析师，查看定时任务与历史产出；打开周末生成的 03_weekly_report.html 即可预览三模型流水线产出的完整可视化周报。
  6. /sessions —— 研究院全部历史产出的全文检索。
  7. /admin/keys —— 运营者签发 sk_ins_* 密钥，让外部系统通过 v1 API 调用研究院。
